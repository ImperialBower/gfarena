# gfcore 0.0.3 — Game History Download & In-Browser Audit

**Date:** 2026-05-10  
**Branch:** audit

## Goal

Upgrade gfarena to gfcore 0.0.3 and surface its new `audit()` feature in two ways:

1. **In-browser audit result** shown in the game-over overlay after each completed game.
2. **Download button** in the score bar that exports the completed game as a `GameCollection` YAML file, compatible with gfcore's offline `audit()` / `replay()` tooling.

## Architecture

Three layers are touched: the gfcore WASM API (new functions), the gfarena Cargo manifest (version bump), and the gfarena frontend (wiring + UI).

### 1. gfcore WASM API (`gfcore/src/wasm_api.rs`)

A new thread-local is added alongside the existing `GAME` and `PROFILES`:

```rust
thread_local! {
    static COLLECTION: RefCell<GameCollection> = RefCell::new(GameCollection::new());
}
```

Four new `#[wasm_bindgen]` functions, all gated on `#[cfg(feature = "history")]`:

| Function | Returns | Purpose |
|---|---|---|
| `collect_game()` | `{"ok":true}` or `{"error":"..."}` | Appends current `GAME` record to `COLLECTION` |
| `clear_collection()` | `{"ok":true}` | Resets `COLLECTION` to an empty `GameCollection` |
| `get_collection_yaml()` | Raw YAML string or `{"error":"..."}` | Serializes `COLLECTION` as `GameCollection` YAML |
| `audit_current_game()` | `AuditResult` JSON or `{"error":"..."}` | Calls `game.record().audit()` and serializes result |

`COLLECTION` is not touched by `new_game`, `new_bot_game`, or `new_human_vs_bots_game` — it is managed entirely by the JS caller.

### 2. Cargo manifest (`gfarena/Cargo.toml`)

```toml
# before
gfcore = { version = "0.0.2", features = ["wasm", "history"] }

# after
gfcore = { version = "0.0.3", features = ["wasm", "history"] }
```

The `[patch.crates-io]` pointing to the local git clone is unchanged.

### 3. Frontend (`gfarena/www/index.html`)

#### Imports

```js
import init, {
  arena_version, new_human_vs_bots_game, get_human_state, act, step_bot,
  collect_game, clear_collection, get_collection_yaml, audit_current_game
} from './pkg/gfarena0_web.js';
```

#### Score bar — Download button

A "Download History" button is added to `#score-bar`, disabled on load:

```html
<button id="download-btn" disabled>Download History</button>
```

Clicking it calls `get_collection_yaml()`, creates a `Blob`, and triggers a download named `gf-history-<unix-timestamp>.yaml`.

#### Game lifecycle changes

`startGame()`:
1. Calls `clear_collection()` — resets the collection.
2. Disables `#download-btn`.

Game-over (inside `showGameOver()` or its call site):
1. Calls `collect_game()` — appends the finished game to the collection.
2. Calls `audit_current_game()` — gets `AuditResult` JSON.
3. Renders audit result into a `#audit-result` div inside `#game-over-box`, positioned between `#final-scores` and `#play-again-btn`:
   - Consistent: a single green "✓ Game record is consistent" line.
   - Violations: a red header "⚠ Audit violations:" followed by each violation as a `<li>`.
4. Enables `#download-btn`.

## Data Flow

```
new game starts
  └─ startGame() ──► clear_collection()          COLLECTION = []
                 ──► download-btn disabled

game plays …
  └─ game over detected (scheduleLoop / showGameOver)
       ├─ collect_game()                          COLLECTION = [GameRecord]
       ├─ audit_current_game() ──► AuditResult JSON
       ├─ render audit result in overlay
       └─ download-btn enabled

user clicks Download History
  └─ get_collection_yaml() ──► YAML blob ──► browser download
                                              filename: gf-history-<ts>.yaml

user clicks Play Again / New Game
  └─ startGame() ──► clear_collection()          COLLECTION = []
```

## Error Handling

- `collect_game()` / `audit_current_game()` errors are logged to `console.error`; the overlay shows "Audit unavailable" rather than crashing.
- `get_collection_yaml()` errors are logged; the download is silently skipped (button stays enabled for retry).
- No user-visible error modals — this is non-critical metadata.

## Out of Scope

- Replay verification (`replay()`) — WASM games do not record `initial_draw_pile`, so replay is not possible.
- Persisting history across page reloads (localStorage / IndexedDB).
- Multi-game accumulation across "Play Again" sessions — collection always holds exactly one game.
